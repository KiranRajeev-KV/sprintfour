# Writeup

For this challenge, I built Redactlane as a batch-first anonymization system rather than a traditional document editor. The core product decision was to optimize for Maya's actual bottleneck: throughput under time pressure. Instead of making her open 200 files one by one, the system ingests a large batch, processes files asynchronously, and immediately separates the safe majority from the exceptions. The dashboard is designed around the questions Maya needs answered quickly: what is ready to approve now, what needs review, what failed, and what is exportable. That led to explicit document states (`QUEUED`, `PROCESSING`, `READY`, `NEEDS_REVIEW`, `CLEAN`, `FAILED`, `APPROVED`, `EXPORTED`), bulk actions for approve and retry, and focused exception handling for the documents that actually need review.

On the backend, I chose Go because this problem is fundamentally about operational throughput, concurrency control, and predictable failure handling more than model experimentation. Go made it straightforward to build a bounded worker pool that can process many documents in parallel, isolate failures, retry safely, and keep the server responsive without adding queue infrastructure. The worker-pool design matters here: one failed document must not block 199 successful ones, and repeated actions need to stay idempotent. This is also why the backend keeps explicit runtime state, retry counts, and export behavior simple and deterministic.

For detection, I intentionally did not make AI the foundation of the product. The default path is regex/rule-based because it is fast, transparent, and reliable for structured identifiers. I added an optional local GLiNER sidecar for semantic labels like person, address, email, and phone, but treated it as an enhancement rather than a dependency. If the sidecar is unavailable or too slow, the system degrades gracefully to regex-only detection instead of stalling the batch. On the review side, I added a focused exception workflow with suggested redactions, manual redaction support, explicit review states, and batch-safe approval and export rules.

Just as important, I chose not to build several things. I did not build PDF-native redaction, OCR, authentication, collaboration, WebSocket infrastructure, or a full audit/compliance layer. I also did not make the review editor the center of the experience. All of those may be useful in a larger product, but for this hackathon they would have increased complexity without improving the main outcome: helping Maya finish a large batch before end of day. I prioritized a system that is operational, resilient, and batch-oriented over one that is broad.

With more time, I would add persisted batch state with SQLite, resume/cancel flows, ZIP export, keyboard-heavy exception review, duplicate clustering across documents, and stronger quality instrumentation comparing regex-only versus regex-plus-NER runs. I would also improve the optional sidecar path with better warmup, model health checks, and more rigorous evaluation of where semantic NER improves recall without hurting throughput. The main judgment call throughout was that this problem is less about building the smartest detector and more about building the right workflow for a high-volume user under real pressure.
